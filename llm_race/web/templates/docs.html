About Benchmarks

How LLM Race measures and reports LLM inference performance.

Overview

LLM Race is a benchmarking tool that measures the performance of Large Language Models across different providers, hardware configurations, and usage patterns. Each benchmark run consists of one or more scenarios — combinations of concurrency level and prompt size that simulate real-world usage. The tool collects per-request timing data and aggregates it into meaningful metrics for comparison.

Workload Profiles

A workload profile defines a set of concurrency levels and prompt sizes that represent a realistic usage pattern. Each profile is designed to stress the model in a different way.

Profile Description Concurrency Prompt Sizes (tokens) Behavior
single-user Single request, measure raw latency 1 64 4096 single request
chat Conversational flow with short context exchanges 1 128 conversational (short user messages)
multi-agent Multiple independent agents running in parallel 2 8 512 2048 independent parallel agents (user queries + tool context)
high-throughput Many users hitting the endpoint simultaneously 32 64 128 64 256 1024 constant concurrent load
stress Maximum concurrency until degradation 256 512 1024 4096 degradation testing
voicebot Voice assistant — short queries, minimal TTFT, no thinking 1 16 32 voice assistant (short query, fast response, no thinking)

Each scenario runs 2 warmup iterations followed by 10 measured iterations to gather timing data. Prompt sizes are defined per-profile in the codebase — see llm_race/bench/workloads.py.

Voicebot Workload

The voicebot workload is designed to find the fastest model and provider for voice assistant applications. Voice assistants require near-instant responses: the user speaks a short query and expects an immediate reply. Any noticeable delay breaks the illusion of a natural conversation.

What makes it different

  • Very small prompts (16–32 tokens) — Voice queries are short: "What's the weather?", "Set a timer for 5 minutes", "Call mom".
  • Short responses — Voice replies should be concise. Use --max-tokens 64 or --max-tokens 128 to simulate real voice output.
  • No thinking mode — Thinking/reasoning tokens add latency without improving voice responses. Disable thinking on your provider for accurate results.
  • Single concurrency — One user, one voice interaction at a time.
  • TTFT is the key metric — Time to First Token determines how quickly the assistant starts speaking. Lower TTFT means a more responsive voice experience.

How to run

Run the voicebot benchmark with a small --max-tokens to simulate short voice responses: 

python3 -m llm_race run --workload voicebot --model <model> --provider <provider> --base-url <url> --max-tokens 64 --temperature 0

Recommended settings for voicebot benchmarks:

  • --max-tokens 64 — Short voice responses (recommended)
  • --temperature 0 — Deterministic output for consistent comparisons
  • Disable any "thinking" or "reasoning" mode on your provider endpoint

The primary metric to compare is TTFT (Time to First Token). A lower TTFT means the model starts generating faster, which translates to a more responsive voice assistant. Secondary metrics include E2E latency and prompt processing speed (PP).

Measured Metrics

Metrics are collected per-request and aggregated across all measured iterations. All latency values are stored in milliseconds in the database and displayed rounded to one decimal place.

⚡ PP (Prompt Processing)

How fast the system processes the input prompt during the prefill phase. Measured as prompt tokens per second until the first output token.

PP = prompt_tokens ÷ TTFT

📊 TGS (Token Generation Speed)

Output tokens per second averaged across all requests in a scenario. The total completion tokens divided by wall-clock time.

TGS = ∑ completion_tokens ÷ wall_clock_seconds

⏱️ TTFT (Time to First Token)

Time between sending the request and receiving the first output token. Measures the initial model latency — prompt processing, prefill, and queue wait.

TTFT = response.first_token_time − request.sent_time

📏 E2E Latency (End-to-End)

Total time from request submission to final token received. The full round-trip including network, queue, prefill, and decode.

E2E = response.last_token_time − request.sent_time

⚡ ITL (Inter-Token Latency)

Time between consecutive output tokens during streaming. Represents the decode speed — how fast the model generates each subsequent token.

ITL = token[i+1].time − token[i].time

🔁 Throughput (RPS)

Requests per second. The number of completed requests divided by the wall-clock time of the scenario.

RPS = successful_requests / wall_clock_seconds

⏳ Wall Clock Time

Real elapsed time for the entire scenario (warmup + measured iterations). This is the actual time the benchmark took to run.

✅ Request Status

Each request is tracked as successful or failed, with error messages recorded for failures. The success rate helps assess reliability under load.

🏷️ Token Counts

Prompt tokens (input), completion tokens (output), and total tokens per request. Reported by the provider or counted client-side.

Percentile Metrics

For E2E, TTFT, and ITL, the benchmark reports multiple percentile values to give a complete picture of the latency distribution:

Metric What it tells you
Mean Average latency — useful for capacity planning and cost estimation.
P50 (Median) Typical user experience — half of requests are faster than this.
P90 90th percentile — most users (9 out of 10) will experience this or better.
P99 99th percentile — the worst-case for all but 1% of requests. Reveals tail latency.
Max Absolute worst-case latency observed. Only reported for E2E.

Providers

LLM Race supports multiple providers through a common interface. Each provider implements stream_complete(prompt) and complete(prompt) and returns token streams with timestamps:

  • vLLM — OpenAI-compatible API, synchronous streaming over HTTP.
  • Ollama — Local Ollama instance via its REST API.
  • LM Studio — Local inference server with OpenAI-compatible endpoint.
  • MLX — Apple Silicon optimized MLX runtime.

All providers use httpx.AsyncClient for concurrent requests. API keys are read from environment variables or a .env file.

Machine Information

Every benchmark records the machine specs to enable fair comparisons:

  • Hostname — machine identifier.
  • CPU — processor model.
  • GPU — GPU model(s) and count.
  • RAM — total system memory in GB.
  • OS — operating system name and version.
  • Driver version — GPU driver (e.g., CUDA version).
  • Python version — runtime version.

Machine info is collected once per session and cached. Pass --force-detect to re-collect.

Database Schema

Data is stored in a local SQLite database at data/benchmarks.db. The key tables are:

  • models — LLM model metadata (name, version, quantization, provider, context window).
  • machines — Hardware specs for the benchmarking machine.
  • benchmarks — Per-run scenario results with all aggregated metrics.
  • results — Per-request raw measurements (timestamps, token counts, status).

How to Run a Benchmark

  1. Start your provider (e.g., vLLM server, Ollama, LM Studio).
  2. Run: python3 -m llm_race run --model "<model>" --provider <provider> --base-url <url>
  3. View results: python3 -m llm_race.web and open the web viewer.

Use --workload to choose a profile, --concurrency and --prompt-lengths for custom scenarios, or --preset to run predefined model/provider combinations from presets.json.
