## Collector Functions Implementation (Task 2) - 2026-06-27

### What was done
- Added imports `os`, `platform`, `re`, `subprocess` to `llm_race/utils/system.py` (stdlib group before `psutil`).
- Implemented five private collector functions before `_cached_info`:
  - `_get_os_info()` — uses `platform` module; on macOS uses `platform.mac_ver()` for user-facing OS version.
  - `_get_cpu_info()` — uses `platform.processor()`, `platform.machine()`, `os.cpu_count()`.
  - `_get_ram_info()` — uses `psutil.virtual_memory()`, returns total/available in GB.
  - `_get_network_info()` — uses `socket.gethostname()` and `socket.gethostbyname()` with fallback to `127.0.0.1`.
  - `_get_gpu_info()` — tries NVIDIA → AMD → Apple Silicon → None fallback.
- Added three GPU-specific helpers (`_get_nvidia_gpu_info`, `_get_amd_gpu_info`, `_get_apple_gpu_info`) with subprocess timeouts and specific exception handling.

### Verification
- Runtime import test: all five functions callable and return expected dict shapes.
- On macOS Apple M4 Pro: GPU detected as `Apple Apple M4 Pro`; `gpu_driver_version`, `gpu_cuda_version`, `gpu_vram_gb` correctly `None` because `system_profiler` output didn't expose VRAM total in the expected format.
- `pytest tests/test_system.py` → 10 passed.
- LSP (`basedpyright`) still not installed; verification done via runtime tests as before.

### Patterns observed
- Subprocess calls need explicit `timeout` to avoid hanging when tools are missing.
- Specific exception tuples (`FileNotFoundError`, `TimeoutExpired`, `CalledProcessError`) keep fallback path clean.
- `re` imports placed inside helpers or at module top both work; module-level import keeps style consistent with stdlib group.

## 2026-06-27: collect_system_info() Implementation

### Implementation Details
- Replaced `NotImplementedError` with actual collector calls in `collect_system_info()`
- Calls: `_get_os_info()`, `_get_cpu_info()`, `_get_ram_info()`, `_get_network_info()`, `_get_gpu_info()`
- Assembles into `SystemInfo` dataclass with all 18 fields
- Wraps exceptions in `SystemInfoError` with context message
- Caches result in module-level `_cached_info`

### Machine Model Mapping (to_dict)
- `SystemInfo.hostname` → `Machine.hostname`
- `SystemInfo.cpu` → `Machine.cpu`
- `SystemInfo.gpu` → `Machine.gpu`
- `SystemInfo.gpu_count` → `Machine.gpu_count`
- `SystemInfo.ram_total_gb` → `Machine.ram_gb` (renamed)
- `SystemInfo.os` → `Machine.os`
- `SystemInfo.os_version` → `Machine.os_version`
- `SystemInfo.gpu_driver_version` → `Machine.driver_version` (renamed)
- `SystemInfo.python_version` → `Machine.python_version`
- `created_at` is auto-generated by SQLAlchemy, not in mapping

### Type Notes
- `_get_cpu_info()` returns `dict[str, str | int]` — type narrowing happens at assignment (platform.processor() always returns str, os.cpu_count() always returns int)
- Machine model fields are all Optional except hostname, so None values are fine for GPU fields

### Verification Results
- `collect_system_info()` returns valid SystemInfo with hostname="mac-mini.local", cpu="arm"
- Caching: `get_system_info(force=True)` returns new instance, second call returns same (True)
- `to_dict()` returns 9 keys matching Machine model fields

### Patterns observed
